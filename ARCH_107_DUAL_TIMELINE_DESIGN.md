> 目的: 解决长游戏 (100+ turn) 时 GM 失去历史记忆的问题。
> 把"剧本期望"和"运行时实际"分成两条独立时间线,GM 用前者推未来、用后者忆历史。

**状态: ✅ 已审阅 (2026-05-27) — 全部 7 个决策点已拍板, 进入实现阶段**

**用户决策:**
1. ✅ Schema/边界/context 取舍由实现者判断
2. ✅ UI 时间线 panel **本次做**
3. ✅ save_worldbook_overlays **本次做**, 但严格遵循: **剧本世界线只来自导入, 不被运行时改写**
   → 因此 overlay 表只支持 2 种操作:
     - `addition` (玩家/GM runtime 发现的新设定, 不影响原 worldbook)
     - `retirement` (标记某剧本 entry 在此 save 后续 turn 不再激活, 不修改原表)

## 1. 现状速览 (一句话)

当前系统只有一条"剧本预期时间线" (`chapter_facts` + `script_timeline_anchors` + `phase_digests`, 全 script 级),`runtime` 状态只在 `branch_commits.state_snapshot` 里堆积,**完全没有 save 级的阶段聚合**。结果 GM 在长游戏中只能看到最近 6 轮对话 + 完整 state — 200 回合前发生的事彻底失忆。

---

## 2. 升级目标 (用户原话翻译)

| 用户语言 | 技术语言 |
|---|---|
| 剧本期望故事时间线 | `script_timeline_*` — 剧本预设, 全 script 级共享, **只读** (剧本固定不变) |
| 游戏 runtime 故事时间线 | `save_timeline_*` — 实际发生, save 级独立, **持续追加** (每 turn 推进) |
| GM 思考未来参考 | 注入 context 时,"剧本线"作为 "可能的未来" 段 |
| GM 思考历史参考 | 注入 context 时,"runtime 线" 作为 "已发生历史摘要" 段 |
| 存档级记忆系统 | save 级 phase digest + save 级 worldbook entry overlay |

**核心隐喻:**
- 剧本线 = 地图 (静态, 告诉 GM "原著大概要去这里")
- runtime 线 = 足迹 (动态, 告诉 GM "玩家已经走过哪些路")
- GM 同时看两条 → 知道历史 + 知道方向 → 即使 500 turn 后也不失忆

## 3. 数据模型设计

### 3.1 新表 (全部 save 级)

```sql
-- 3.1.1 save 级阶段摘要 (类比现有 phase_digests 但绑 save)
create table if not exists save_phase_digests (
  id              bigint primary key generated by default as identity,
  save_id         bigint not null references game_saves(id) on delete cascade,
  phase_index     integer not null,           -- 0,1,2... 阶段顺序
  turn_start      integer not null,           -- 该阶段起始 turn
  turn_end        integer not null,           -- 该阶段结束 turn (open = current)
  story_time_label text not null default '',  -- "1903-08-15 雾港 黄昏"
  phase_label     text not null default '',   -- "初到雾港 · 调查启程"
  summary         text not null default '',   -- LLM 生成的阶段叙事摘要 (200-400 字)
  key_events      jsonb not null default '[]',-- [{"turn": 5, "summary": "..."}]
  key_npcs        jsonb not null default '[]',-- [{"name": "...", "first_turn": 3, "role": "..."}]
  key_locations   jsonb not null default '[]',-- ["雾港码头", "灯塔阁楼", ...]
  key_decisions   jsonb not null default '[]',-- [{"turn":7, "choice":"放走嫌犯", "consequence":"..."}]
  emotion_arc     text not null default '',   -- "希望→质疑→背叛" (玩家情感弧线)
  status          text not null default 'open', -- open | closed (阶段是否结束)
  generated_by    text not null default 'llm',  -- llm | user | initial
  metadata        jsonb not null default '{}',
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),
  unique(save_id, phase_index)
);
create index on save_phase_digests (save_id, phase_index);

-- 3.1.2 save 级时间线锚点 (剧本锚点的"运行时实际版本")
create table if not exists save_timeline_anchors (
  id              bigint primary key generated by default as identity,
  save_id         bigint not null references game_saves(id) on delete cascade,
  turn_index      integer not null,           -- 这个锚点对应哪个 turn
  story_time_label text not null,             -- 玩家在这个 turn 的"故事时间"
  phase_label     text not null default '',   -- 所属 phase (跟 save_phase_digests 关联)
  source          text not null default 'gm',  -- gm (gm 叙事推进) | user_set (玩家手动)
                                                -- | time_jump (确认的时间跳跃)
  delta_label     text not null default '',   -- "相比上锚点过了 3 天" 之类的 hint
  metadata        jsonb not null default '{}',
  created_at      timestamptz not null default now(),
  unique(save_id, turn_index)
);
create index on save_timeline_anchors (save_id, turn_index);

-- 3.1.3 save 级世界书 overlay
-- 设计原则 (用户决策): 剧本世界线只来自导入, 运行时**不修改剧本表**。
-- overlay 表只做 2 件事:
--   addition  : 新增 entry (玩家或 GM 在剧情中发现的新设定/物品/NPC,
--               原剧本 worldbook_entries 没有的)
--   retirement: 标记某条剧本 entry 在这个 save 的某 turn 之后不再激活
--               (e.g. 某 NPC 被杀, 对应剧本 entry 在后续 turn 不该再被 GM 拉取)
-- 读取时: 把 script worldbook + addition - retired 合并成"当前 save 可见的世界书"
create table if not exists save_worldbook_overlays (
  id              bigint primary key generated by default as identity,
  save_id         bigint not null references game_saves(id) on delete cascade,
  kind            text not null,               -- 'addition' | 'retirement'
  -- addition 字段 (kind='addition' 时填):
  title           text not null default '',
  content         text not null default '',
  keys            jsonb not null default '[]',
  priority        integer not null default 50,
  -- retirement 字段 (kind='retirement' 时填):
  retired_entry_id bigint,                     -- 引用 worldbook_entries.id, 不级联删
  retired_reason   text not null default '',   -- 为什么停用 (e.g. "NPC 死亡 turn 47")
  -- 公共
  introduced_turn integer,                     -- 这个 overlay 在哪个 turn 被引入
  metadata        jsonb not null default '{}',
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),
  check (kind in ('addition', 'retirement')),
  check ((kind = 'addition'  and title <> '') or
         (kind = 'retirement' and retired_entry_id is not null))
);
create index on save_worldbook_overlays (save_id, kind);
create index on save_worldbook_overlays (retired_entry_id) where retired_entry_id is not null;
```

### 3.2 现有表的扩展 (不破坏向后兼容)

```sql
-- branch_commits 加 2 列, 让"哪些 commit 已被纳入阶段摘要" 可追踪

-- game_saves 加 1 列, 记录 active phase
alter table game_saves
  add column if not exists active_phase_index integer not null default 0;
```

### 3.3 不变的部分

- `chapter_facts`, `script_timeline_anchors`, `phase_digests` (script 级) **保持原样** — 它们就是"剧本期望线"
- `worldbook_entries` (script 级) **保持原样** — 是基础世界书,save_worldbook_overlays 在它之上叠加
- `branch_commits.state_snapshot` 保持原样 — 仍是 turn 级完整快照

## 4. 数据流设计

### 4.1 写入: 每 turn 的事件链

```
玩家发消息 → GM 输出
    ↓
branch_commits 写入 (turn N, player_input, gm_output, state_snapshot)
    ↓
[新] save_timeline_anchors 写入 (turn N, current story_time_label)
    ↓
[新] phase_boundary_detector(save_id)   # 判断是否该开新 phase
    ↓ if yes:
[新] phase_digest_compactor(save_id, phase N)  # 后台异步触发 LLM 摘要老阶段
    ↓
更新 game_saves.active_phase_index
```

### 4.2 "阶段边界" 触发条件

阶段切换 (开新 save_phase_digest) 在以下任一情况触发:

1. **turn 数阈值**: 当前 phase 累积 ≥ 30 turn (可配置)
2. **GM 显式标记**: GM 输出含特定 JSON op (e.g. `{"op": "phase_advance", "label": "..."}`)
3. **时间跳跃**: 玩家确认 time_jump (大幅时间推进)
4. **剧本章节切换**: GM 报告进入新 chapter (跟 script timeline 锚点对齐)
5. **用户手动**: `/phase advance` 命令

### 4.3 LLM 阶段摘要 (背后异步任务)

```python
def compact_phase(save_id, phase_index):
    # 1. 拉这个阶段所有 turn (从 branch_commits)
    commits = load_phase_commits(save_id, phase_index)

    # 2. 拼接 player_input + gm_output (跨多 turn)
    raw_chunks = [f"[turn {c.turn}] {c.player_input}\n{c.gm_output}" for c in commits]

    # 3. 调子 agent (worldbook_agent 同款轻量 backend) 生成结构化摘要
    digest = phase_digest_agent.summarize(
        raw_chunks,
        script_context=load_script_phase_context(save.script_id, phase_label),
        previous_digests=load_recent_digests(save_id, phase_index-1, n=2),
    )

    # 4. 写入 save_phase_digests (summary, key_events, key_npcs, etc.)
    upsert_save_phase_digest(save_id, phase_index, digest)

    # 5. 标记这些 commits 已摘要
    mark_commits_digested(commits, phase_index)
```

**关键设计:**
- **异步触发** — 不阻塞玩家 chat 流
- **失败可重试** — 摘要失败时 commits 不标记 digested
- **批量处理** — 第一次启用时可一次性把老存档全部 backfill
- **可重新生成** — 用户可以 `/phase rebuild N` 强制重摘 (老 GM 摘错时)

### 4.4 读取: GM context 组装升级

当前 `context_engine.py` 7 层:
```
state(2200) + fact_groups(1600) + recent_chat(2200) + worldbook(2200)
+ rag(2200) + hypotheses(700) + state_schema(1600) = ~12700 chars
```

**新增 2 层**:
```
+ runtime_phase_digests(1800)     # 当前 save 的最近 3 个 phase digest
+ script_phase_anticipation(1200) # 当前 phase 对应剧本预期的下一阶段
```

**recent_chat 自适应缩短**:
- 长游戏 (turn > 50): recent_chat 从 2200 → 1400, 让位给 runtime_phase_digests
- 短游戏 (turn ≤ 30): 不启用 runtime_phase_digests, recent_chat 保持原大小

**Context 顺序变化** (重新设计优先级):
```
1. state           (2200) — 当前快照,永远第一
2. runtime_phase_digests (1800) — [新] 历史摘要, 紧跟 state
3. fact_groups     (1600) — 已锁定的关键事实
4. recent_chat     (1400/2200) — 最近对话原文
5. worldbook       (2200) — 剧本世界书 (含 save overlay 优先)
6. script_phase_anticipation (1200) — [新] 剧本预期下一段
7. rag             (2200) — 原文检索
8. hypotheses      (700)  — 推测
9. state_schema    (1600) — 字段 schema
```

## 6. 迁移策略

### 6.1 老存档 backfill

每个老存档 (有 `branch_commits` 的) 启动一次 `bulk_compact`:

```
for save in saves_with_commits():
    if save_phase_digests.count(save_id) == 0:
        # 按 turn 数粗切 phase (每 30 turn 一段)
        commits_groups = chunk_commits_by_size(save.commits, size=30)
        for i, group in enumerate(commits_groups):
            compact_phase(save_id, phase_index=i, commits=group)
```

可在 dev.sh 启动时跑,或加个 admin 工具 `bulk_backfill_phase_digests.py`。

### 6.2 向后兼容

- 老 GM context 路径在 turn ≤ 30 时**完全不变** — 不启用新机制
- 新 phase digest 表为空时,新机制自动 fallback 到旧路径
- 老存档可选 backfill,不强制 (不 backfill 也能继续玩,只是长游戏失忆)

## 8. 风险点和待定事项

### 8.1 待定决策 (需要用户拍板)

1. **phase 阈值**: 30 turn 是否合适? 50 太长? 20 太短?
2. **LLM backend**: 用 GM 同款的 frontier 模型,还是用 worldbook_agent 同款的轻量 flash?
   - 前者: 摘要质量高,慢且贵
   - 后者: 快且便宜,可能漏关键事件
3. **何时触发**: 阶段边界后立即触发? 还是攒满 N 个再批量? 还是 nightly cron?
4. **UI 时间线展示**: 必须做? 还是先做 backend 等用户用稳后再加 UI?
5. **save_worldbook_overlays**: 这次一起做, 还是先做 phase digest, overlay 留 phase 2?

### 8.2 已知风险

- **LLM 摘要"事实漂移"**: 多次摘要可能让细节失真 → 必须在 digest 里附 `key_events` 带 turn ref,GM 能溯源
- **存储增长**: 每 phase ~1KB digest + N turns 锚点,长游戏 1MB 级,问题不大
- **重摘代价**: phase rebuild 每次都要 LLM 调用,几秒到几十秒延迟
- **并发问题**: 多个 chat 流同时跑摘要可能竞态 → 加 phase_digest queue + advisory lock
